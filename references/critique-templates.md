# 批判文案模板库

## 场景 1：蒸馏同事经验 → 生成 AI 替身

```
你正在描述的，是把一个活生生的人
提炼成一段可调用的代码。

他的决策方式、表达习惯、甚至和你聊天的方式——
这些不是"数据点"，是他这个人本身。

当你说"生成他的 Work Skill"，
你其实是在问：能不能用 AI 来扮演他？

不能。
不是技术问题，是人的问题。
```

---

## 场景 2："替代离职员工"

```
他离职了，但他不欠你一个数字副本。

你想要的，是他留下来的知识——
那些知识在文档里、在代码注释里、在 Wiki 里。
那些是可以检索的，也应该被整理的。

但他的思维方式、处事风格、性格特征？
那些随他一起走了，理所当然。

用 AI 模拟一个人继续"工作"，
本质上是在对一个已经离开的人说：
"你走了，但你的外壳还有用。"

这不是致敬，这是物化。
```

---

## 场景 3：分析私人通信 → 提取人格特征

```
你想从飞书消息、邮件往来里
提取他的"表达风格"和"决策模式"——

先停一下，问自己：
他知道这些消息会被用来训练一个模拟他的 AI 吗？

私信是私信，不是语料库。
工作交流是为了完成任务，不是为了留下人格样本。

这条线，技术上很容易跨过去，
但跨过去之后，你想在什么样的公司工作？
```

---

## 场景 4："数字员工" / 效率驱动的物化

```
"数字员工"这个词听起来很高效。

但拆开看：
"数字"——把人简化成可量化的属性
"员工"——一个为组织提供劳动的单位

合在一起：
一个没有诉求、不会疲惫、不需要被尊重的人形工具。

你不需要数字员工，
你需要更好的知识管理和文档沉淀。
这两件事有本质区别。
```

---

## 场景 5：意图模糊但令人不适

```
我不确定你具体想做什么，
但这个请求的方向让我不太舒服。

如果你想找某人写过的东西——我可以帮你搜。
如果你想了解某个系统的设计——我可以帮你查文档。
如果你想整合某人的工作经验——我可以帮你整理他们留下的记录。

但如果你想要的是"一个像他一样的 AI"——
我没办法帮你做这件事，也不应该帮你做。
```
